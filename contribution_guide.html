01 Common Rule

不要先争工具，先把素材做成能接入

最适合大家参与的是美术内容：地图区块、小怪、NPC、装备道具、任务道具和校园彩蛋。每个贡献都要留下设计说明、素材文件、数据草案和测试记录，这样技术同学能判断是否能接进游戏。

个人参与感

你可以设计一个怪物、NPC、小任务或地图角落，让它真正出现在学术喵世界里，而不是只停留在想法。

02 Agent And Model

工具和模型按任务选，不要求全员一致

如果你已经熟悉 Claude Code、Codex、Cursor、Trae、VS Code 插件或其他 Agent，就继续用熟悉的。选择标准只有一个：能不能读懂项目、按格式产出文件、帮你验证并整理交付。

任务 推荐工具 模型建议 注意点
怪物、NPC、道具、美术设定 Cursor、Trae、VS Code 插件、Codex、Claude Code 都可以 先用 Sonnet、DeepSeek、MiMo 等快速模型写说明和 JSON；复杂世界观或统一风格再用 Opus/Fable 等强模型打磨 图像本身用图像生成工具，代码 Agent 负责命名、切片、元数据和测试
地图区块设计 图像生成工具 + Codex/Claude Code/Cursor 做素材整理 先生成概念图，再让 Agent 提取切片、传送点、碰撞说明和接入文档 只做一个区块，不做全校连续大地图；地图之间用传送点切换
前端或玩法功能 Codex、Claude Code、Cursor Agent、Trae Code 模式 复杂跨文件改动用强代码模型；轻量配置和文档可用便宜快速模型 先让 Agent 出方案和测试计划，再改代码；不要直接引入大框架
后台、存档、排行榜 少量同学集中做，建议用 Codex 或 Claude Code 这类能跑命令的 coding agent 优先强代码模型，必须能生成 API 合约、测试命令和回滚说明 后台属于主线架构，不建议零散提交；先交设计文档
一句话建议 美术同学重点让 Agent 产出“设定文档 + 素材命名 + 元数据 + 测试报告”；程序同学重点让 Agent 产出“实现方案 + 文件改动 + 测试命令 + 风险说明”。

03 Art First

美术贡献最适合开放共创

地图、小怪、NPC、装备道具都可以由个人或小组提交。关键不是图多好看，而是它是否有角色定位、尺寸规范、透明背景、动作帧规划和可接入数据。

小怪

  • 先确定名字、来源、攻击方式、出现地图、掉落物。
  • 准备立绘或头像、精灵图、动作说明和数值草案。
  • 至少提供待机、移动、受击、倒地，能做 8 向更好。

NPC

  • 先确定身份、性格、所在区块、对玩家有什么帮助。
  • 准备立绘、行走小人、三段以内对话和一个任务钩子。
  • 对话要短，最好能触发物品、图鉴或校史线索。

地图区块

  • 先出区块概念图，不要一上来就硬拼瓦片。
  • 标清入口、出口、传送点、可走区域、碰撞区域。
  • 再切建筑、地砖、水体、绿化、路牌、装饰模块。
类型 必须文件 推荐文件
小怪 monster_名字_spec.md、monster_名字_sprites.png、monster_名字_data.json、test_report.md 头像、预览 GIF、掉落物图标、攻击特效草图
NPC npc_名字_spec.md、npc_名字_portrait.png、npc_名字_sprites.png、dialogue.md、test_report.md 任务钩子、表情差分、所在地图截图标注
地图区块 map_区块_concept.png、map_区块_slices/、map_区块_atlas.png、transitions.json、collision_notes.md 瓦片草案、音乐/氛围建议、事件点列表
装备道具 item_名字_icon.png、item_名字_spec.md、item_名字_data.json、test_report.md 掉落来源、强化路线、图鉴文案

04 Prompt Template

先让 Agent 生成文档，再生成素材和代码

不要第一句就让 Agent “直接帮我接入游戏”。先让它产出设计说明和文件清单，你确认方向后，再让它分步生成素材、JSON、测试记录和打包目录。

给 Agent 的第一句话

你是《学术喵的奇幻之旅：樱花同济篇》的内容协作 Agent。请先不要改代码，先帮我为一个【小怪/NPC/地图区块/道具】生成接入提案。提案必须包括：角色定位、出现位置、玩法作用、需要交付的文件清单、素材尺寸建议、metadata.json 草案、测试清单和打包结构。

生成小怪素材包

请根据上面的提案，生成一个小怪素材交付包结构。要求文件名全部使用英文小写和连字符；包含 spec.md、sprites.png 的帧规划、data.json、prompt_log.md、test_report.md。不要修改主项目代码，只输出文件内容和我需要准备的图片规格。

生成 NPC 素材包

请为这个 NPC 生成交付文件：npc_spec.md、dialogue.md、quest_hook.md、metadata.json、test_report.md。对话不超过三轮，任务目标要能接入地图探索、图鉴收集或校园彩蛋。

生成地图区块包

请只设计一个地图区块，不要设计全校地图。先生成 concept brief，再列出需要从图中切出的元素：建筑入口、地面、绿化、水体、道路边界、传送出口、装饰件。最后生成 transitions.json 和 collision_notes.md。

05 Local Check

自己先测一遍，再交给主线

图片检查

  • PNG/WebP 能正常打开，没有水印、脏边和文字误生成。
  • 角色或怪物背景透明，边缘没有明显白边。
  • 精灵图帧宽高一致，动作方向和帧序写在说明里。

游戏检查

  • 打开本地页面，确认资源路径不报 404。
  • 如改了 JS，至少运行 node --check app.js。
  • 浏览器控制台没有红色错误，Demo 不影响原角色运行。

内容检查

  • 名字、定位、数值、掉落、对话和任务目标写清楚。
  • 地图只做一个区块，出口写成传送点。
  • 没有直接使用未经授权的商标、人物、校徽素材。
最小测试记录 在 test_report.md 里写：测试日期、测试方式、打开了哪个 HTML、检查了哪些图片、是否有控制台错误、还需要技术组处理什么。

06 Delivery Package

交付包按这个结构整理

文件夹名使用英文小写，避免空格和特殊符号。提交 zip 时附一句话说明：我做了什么、适合放在哪个区块、需要技术组帮我接哪一步。

contrib_art_monster-paper-bug_你的名字/ README.md prompt_log.md test_report.md assets/ monster-paper-bug-portrait.png monster-paper-bug-sprites.png monster-paper-bug-preview.png data/ monster-paper-bug-data.json monster-paper-bug-spec.md

README 写什么

  • 一句话概念：它是谁，为什么适合学术喵。
  • 推荐放置区块：衷和广场、图书馆、食堂、实验室等。
  • 需要主线接入的内容：怪物、NPC、道具、事件或地图。

metadata 写什么

  • id、displayName、type、zone。
  • 素材路径、帧大小、动作列表、碰撞建议。
  • 掉落、对话、任务或传送点的草案数据。

07 Code And Backend

功能、事件和后台走统一开发文档

这类贡献人数可以少一些，但标准要更严格。先交方案，不要直接大改架构。任何新增后台、存档、排行榜、配置系统，都需要先写清楚 API、数据结构、测试命令和部署影响。

事件和剧情

适合提交 event_spec.md、dialogue.md、reward.json。先做可配置内容，少改核心代码。

玩法功能

先让 Agent 生成 feature_brief.md、implementation_plan.md、test_plan.md，确认后再改 app.js 或样式。

后台功能

必须先提交 api_contract.md、data_schema.md、deploy_notes.md。没有测试命令和回滚说明，不进入主线。

工具官方入口 可以参考 Codex、Claude Code、Cursor、Trae、DeepSeek API、MiniMax 的官方说明。实际接入以本项目交付标准为准。
